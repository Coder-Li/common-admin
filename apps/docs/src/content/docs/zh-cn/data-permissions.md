Common Admin 的数据权限基础控制操作者可以通过按部门限定的可见性规则查看或分配哪些用户。

在 starter 中，这不是独立的公开 CRUD 资源。它是供用户和组织工作流使用的服务边界。

## 当前边界

数据权限服务会解析当前用户的权限上下文，并为用户记录构建 Prisma 可见性条件。

支持的可见性概念包括：

- 所有用户；
- 自身可见性；
- 按部门限定的可见性；
- 没有允许范围时无可见用户。

该服务还会在创建或更新用户时检查操作者是否可以分配目标部门。

## 何时使用

当管理操作应受到操作者允许的组织范围限制时，请使用数据权限检查，例如：

- 列出用户；
- 读取用户详情；
- 更新用户；
- 为用户分配部门；
- 未来需要基于部门可见性的资源。

不要把前端筛选当作安全边界。后端服务必须在返回数据或接受变更前应用可见性条件。

## 与 RBAC 的关系

RBAC 回答：“这个用户是否可以执行此操作？”

数据权限回答：“这个用户可以操作哪些记录？”

两者可以同时适用于同一个端点。例如，某个用户可能拥有 `user.update`，但仍然不能更新其数据范围之外的目标用户。

## 实现规则

扩展数据权限行为时：

- 保持权限上下文解析集中化；
- 构建一个由 list 和 count 查询共享的可见性条件；
- 当调用者不应知道记录是否存在时，对可见性之外的记录返回 `404`；
- 根据操作者允许的部门范围校验部门分配；
- 测试空范围行为；
- 避免通过选项端点、导出或计数泄露隐藏记录。

## 前端规则

前端页面可以基于可用选项显示筛选器或选择器，但不应复制后端可见性逻辑。

使用生成的 API helper，并根据后端响应展示普通的空、禁止访问或未找到状态。

## 验证

聚焦检查：

```bash
pnpm --filter api test -- data-permission
pnpm --filter api test -- user
```

当可见性影响 HTTP 行为时：

```bash
pnpm --filter api test:e2e
pnpm build
```
